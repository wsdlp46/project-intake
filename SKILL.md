# 项目材料入库

工作区标准见 `<workspace>/01-通用规则/项目工作流规范.md`（`<workspace>` 为用户配置的工作区根目录）。本 Skill 执行规范中的阶段一（材料入库）。

## 前置条件

用户需指明要入库到哪个项目（如"双碳平台"）。若未提供，先问"入库到哪个项目？"。

## 工作流：五步入库

### 步骤1：判断类型

对每个文件/内容，判断：

| 类型 | 识别信号 | 入库动作 |
|------|---------|---------|
| PDF | .pdf | 归档 _source/ → 执行转换（步骤2） |
| DOCX | .docx | 同上 |
| XLSX | .xlsx | 同上 |
| PPTX | .pptx | 同上 |
| 会议记录 | 用户口述/粘贴内容，或文件名含"会议""沟通""纪要" | 提取时间/来源/内容（步骤3） |
| 需求/合同类 | 文件名含"合同""需求""招标""SOW" | 存放 01-项目输入/，标记为原始输入 |

### 步骤2：二进制转换

适用于 PDF/DOCX/XLSX/PPTX。具体转换流程见工作流规范第四章。

关键规则：
- 原始文件 → 项目根 `_source/`（不存在则新建）
- 转换产物 `{原文件名}_转换.md` → 与源文件同在 `_source/` 内
- 转换后更新项目 `_index.md` 的"最近更新"（项目文件）或对应子目录 `_index.md`（知识库文件）

### 步骤3：会议时间线更新

适用于会议记录。读取或追加 `01-项目输入/会议纪要_时间线.md`，新增条目格式：

```markdown
## YYYY-MM-DD | [来源/谁] [会议/文档名]

**核心内容**：（3句话概括）

**对项目的影响**：

**关联**：[[原始纪要路径]]
```

**同步归档原始记录**：如果是完整的 MD 格式会议文档，同时保存到 `01-项目输入/会议纪要/原始纪要/[日期]-[主题].md`（目录不存在则新建）。

### 步骤4：关键决策提炼

扫描步骤1-3的内容，识别关键决策信号：

| 信号词 | 触发 |
|--------|------|
| "确定"、"决定"、"定下来" | 提取技术选型/范围调整结论 |
| "不做了"、"砍掉"、"不在范围" | 提取范围约束 |
| "甲方说"、"客户要求" | 提取外部约束 |

如果检测到决策信号，草拟 `#Dxxx` 条目追加到 `01-项目输入/关键决策.md`，标注为待确认。格式：

```markdown
### #DXXX | YYYY-MM-DD | [草拟标题]

**依据**：
**影响范围**：
**关联输入**：
**状态**：待确认
```

**禁止自动写入已确认状态。** 必须标注"待确认"，等用户审阅后手动修改状态。

### 步骤5：更新索引

更新项目顶层 `00-项目总览.md` 的"最近更新"和 `_index.md`：
- `00-项目总览.md`：新增 `- YYYY-MM-DD | 入库 [文件名]`
- `_index.md`：文件清单追加新条目、"最近更新"追加一行

涉及子目录的，同步更新子目录 `_index.md`。

## 输出规范

入库完成后，输出三段摘要：

```
[项目名] 入库完成

文件：
- [文件名] → [存放位置]
- [文件名] → [存放位置] + _转换.md

提炼：
- 会议时间线新增 X 条
- 关键决策草拟 #Dxxx（待确认）

影响提醒：
- [如有影响现有需求的条目，在此标注]
```

---

## 通用规则

- 转换命令使用系统 Python `/usr/bin/python3`（避免 managed Python 的签名问题）
- 文件路径中的中文和空格直接使用，不要转义
- 编号 #Dxxx 确保不重复，读取已有决策中的最大编号再 +1
- 入库过程中任何错误，输出问题到摘要的"影响提醒"栏而非中断流程
